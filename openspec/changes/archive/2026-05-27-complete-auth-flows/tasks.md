## 2. Token plumbing

- [x] 2.1 Implement `src/lib/auth/tokens.ts` exporting `issueToken(user_id, purpose)` (returns `{ token, hash, expires_at }` where `token` is 32 random bytes hex-encoded and `hash = sha256(token)`) and `consumeToken(plaintext, purpose)` (looks up by hash, throws `ApiError("RESOURCE_NOT_FOUND")` on miss and `ApiError("BUSINESS_RULE_VIOLATION")` on expired/consumed, otherwise sets `consumed_at` and returns `{ user_id }`). Behavior delivered: this is the single chokepoint that backs both reset-password rejection (covers "Reset-password rejects unknown, expired, and consumed tokens") and verify-email rejection (covers "Verify-email rejects unknown and expired tokens"). Verify: Jest unit test in `src/lib/auth/tokens.test.ts` covers the four cases — fresh token resolves to its user, unknown hash throws `RESOURCE_NOT_FOUND`, expired-but-unconsumed throws `BUSINESS_RULE_VIOLATION`, consumed throws `BUSINESS_RULE_VIOLATION` — using a Prisma test DB or a thin in-memory shim.
- [x] 2.2 Implement `src/lib/auth/emails.ts` with `sendPasswordResetEmail(to, token)` and `sendVerifyEmail(to, token)` that render plaintext + minimal HTML bodies linking to `${NEXT_PUBLIC_APP_URL}/auth/reset?token=...` and `${NEXT_PUBLIC_APP_URL}/auth/verify?token=...` respectively, and dispatch via `notify.sendEmail`. Behavior delivered: callers anywhere in the codebase can issue these two emails without knowing the link or subject format. Verify: Jest unit test mocks `notify.sendEmail`, calls each helper with a fixture token `"deadbeef"`, and asserts the captured payload includes a body substring matching the expected link template and a non-empty `subject`.
- [x] 2.3 Extend `issueToken` in `src/lib/auth/tokens.ts` to set `consumed_at = now` on every prior unconsumed row of the same purpose for the same user before inserting the new row. Behavior delivered: only the most recently issued link is ever valid, satisfying "Re-issuing a reset token supersedes prior unconsumed reset tokens" and "Re-issuing a verification token supersedes prior unconsumed verification tokens". Verify: Jest unit tests in `src/lib/auth/tokens.test.ts` assert (a) issuing a second reset token for the same user marks the first row's `consumed_at` non-null while the second remains null and the first token's `consumeToken` call throws `BUSINESS_RULE_VIOLATION`, (b) issuing a token for user B does not touch user A's prior unconsumed rows, (c) issuing a verify token does not touch the same user's `PasswordResetToken` rows.

## 3. NextAuth session tracking

- [x] 3.1 Land the "Persisting sessions without leaving the JWT strategy" decision: in `src/lib/auth.ts`, add an `events.signIn` handler that creates a `Session` row on credentialed sign-in (with `refresh_token_hash` = SHA-256 of a fresh 32-byte secret, `expires_at` = now+30d, `ip`/`user_agent` from the request, `last_seen_at` = now), and extend the `jwt` callback to embed the new row's id as a `sid` claim. Behavior delivered: every successful credentials sign-in produces exactly one new `Session` row and a JWT cookie whose payload includes `sid` (satisfies "Credentialed sign-in persists a Session row"). Verify: manually sign in on the dev server with a test credentials account, decode the `next-auth.session-token` cookie payload at `https://jwt.io` (or via a one-off `node -e` script), and confirm a `sid` claim is present whose value matches a freshly inserted `Session.id` for that user.
- [x] 3.2 Extend NextAuth's `session` callback in `src/lib/auth.ts` to look up `Session` by the JWT's `sid` claim and return an empty session (so `auth()` resolves to a session whose `user` is undefined) when the row is missing, when `revoked_at` is non-null, or when `expires_at <= now`; bump `last_seen_at` to now only if more than 60 seconds have passed since the previous value. Treat a missing `sid` claim as a legacy JWT — pass through without a DB lookup so existing logged-in users are not stranded on deploy. Behavior delivered: revoking a row immediately logs the holder out on their next request (satisfies "Authenticated requests are rejected when their session is revoked or expired"). Verify: manual test — sign in, then `UPDATE "Session" SET revoked_at = now() WHERE id = '<sid>'` via psql, refresh any authenticated page, confirm the response treats the user as logged out (e.g. `requireUser` throws / a `/api/auth/me` call returns unauthorized).

## 4. Account recovery routes

- [x] 4.1 Land the "Forgot-password against enumeration" decision and implement `POST /api/auth/forgot-password` at `src/app/api/auth/forgot-password/route.ts`. On a known active email, call `issueToken(user.id, "reset")` and `sendPasswordResetEmail`. On an unknown email, run a dummy `bcrypt.compare` against a fixed precomputed hash and emit a `console.info("[forgot-password:no-match]", ...)` log. Always respond `200 { data: {} }`. Behavior delivered: satisfies "Forgot-password issues a single-use reset token" for matches and "Forgot-password does not leak account existence" for misses. Verify: Jest integration test (or supertest-style harness against `route.ts`) seeds one user, sends the request twice — once with that email, once with a never-registered email — and asserts: response body and status are byte-identical, exactly one `PasswordResetToken` row exists scoped to the known user, zero `sendEmail` calls happen for the unknown case, and the `[forgot-password:no-match]` log line is emitted for the unknown case.
- [x] 4.2 Implement `POST /api/auth/reset-password` at `src/app/api/auth/reset-password/route.ts`. Validate `new_password` against `z.string().min(8).max(100)`. In a single `prisma.$transaction`, call `consumeToken(plaintext, "reset")`, `bcrypt.hash(new_password, 12)` → set on `User.password_hash`, and `prisma.session.updateMany({ where: { user_id, revoked_at: null }, data: { revoked_at: new Date() } })`. Behavior delivered: satisfies "Reset-password rotates the password and revokes sessions" and "Reset-password rejects unknown, expired, and consumed tokens". Verify: Jest integration test walks the full failure matrix in `specs/account-recovery/spec.md` (unknown hash → 404; consumed → 400; expired → 400; weak password → 400 VALIDATION_ERROR; valid → 200) and on the success branch additionally asserts (a) bcrypt now verifies against the new password but not the old, (b) the matching token row has `consumed_at` set, (c) all of the user's prior `revoked_at: null` sessions now have `revoked_at` set.

## 5. Email verification

- [x] 5.1 Hook `POST /api/auth/register` (`src/app/api/auth/register/route.ts`) so that after `prisma.user.create` succeeds (both personal and company branches), it calls `issueToken(user.id, "verify")` and `sendVerifyEmail(user.email, token)`, wrapped in a try/catch that logs failures but does not roll back the user. Behavior delivered: satisfies "Registration issues a verification token". Verify: Jest integration test (a) registers a new user, asserts response is `200` and exactly one `EmailVerificationToken` row exists for that user with `consumed_at` null and `expires_at` ~24h out; (b) registers a second user with `notify.sendEmail` mocked to throw, asserts response is still `200`, the user row exists, the token row exists, and a server log records the dispatch failure.
- [x] 5.2 Implement `GET /api/auth/verify-email` at `src/app/api/auth/verify-email/route.ts`. Hash the `token` query param, look up the matching `EmailVerificationToken`. Branches: (a) no row → `404 RESOURCE_NOT_FOUND`; (b) row exists, user already verified → `200 { data: { user_id, already_verified: true } }` regardless of `consumed_at`/`expires_at` (satisfies "Re-verification is idempotent"); (c) row exists, user not verified, `expires_at <= now` → `400 BUSINESS_RULE_VIOLATION`; (d) row exists, user not verified, unconsumed, unexpired → in one transaction set `User.email_verified_at = now` and `token.consumed_at = now`, return `200 { data: { user_id, already_verified: false } }`. Behavior delivered: satisfies "Verify-email link sets `email_verified_at`" and "Verify-email rejects unknown and expired tokens" alongside the idempotency requirement. Verify: Jest integration test walks the four matrix rows in `specs/email-verification/spec.md` and asserts each response status, body, and DB-mutation outcome.
